  57.  57 
  58.  58 <h1>Security</h1>
  59.  59 
  60.  60 <p>This page describes some "best practices" regarding web security, and details
  61.  61 CodeIgniter's internal security features.</p>
  62.  62 
  63.  63 
  64.  64 <h2>URI Security</h2>
  65.  65 
  66.  66 <p>CodeIgniter is fairly restrictive regarding which characters it allows in your URI strings in order to help
  67.  67 minimize the possibility that malicious data can be passed to your application.  URIs may only contain the following:
  68.  68 </p>
  69.  69 
  70.  70 <ul>
  71.  71 <li>Alpha-numeric text</li>
  72.  72 <li>Tilde: ~ </li>
  73.  73 <li>Period: .</li>
  74.  74 <li>Colon: :</li>
  75.  75 <li>Underscore: _</li>
  76.  76 <li>Dash: -</li>
  77.  77 </ul>
  78.  78 
  79.  79 <h2>Register_globals</h2>
  80.  80 
  81.  81 <p>During system initialization all global variables are unset, except those found in the $_GET, $_POST, and $_COOKIE arrays. The unsetting
  82.  82 routine is effectively the same as register_globals = off.</p>
  83.  83 
  84.  84 <a name="error_reporting"></a>
  85.  85 <h2>error_reporting</h2>
  86.  86 
  87.  87 <p>
  88.  88     In production environments, it is typically desirable to disable PHP's
  89.  89     error reporting by setting the internal error_reporting flag to a value of 0. This disables native PHP
  90.  90     errors from being rendered as output, which may potentially contain
  91.  91     sensitive information.
  92.  92 </p>
  93.  93 
  94.  94 <p>
  95.  95     Setting CodeIgniter's <kbd>ENVIRONMENT</kbd> constant in index.php to a
  96.  96     value of '<kbd>production</kbd>' will turn off these errors. In development
  97.  97     mode, it is recommended that a value of '<kbd>development</kbd>' is used.
  98.  98     More information about differentiating between environments can be found
  99.  99     on the <a href="environments.html">Handling Environments</a> page.
  100. 100 </p>
  101. 101 
  102. 102 <h2>magic_quotes_runtime</h2>
  103. 103 
  104. 104 <p>The magic_quotes_runtime directive is turned off during system initialization so that you don't have to remove slashes when
  105. 105 retrieving data from your database.</p>
  106. 106 
  107. 107 <h1>Best Practices</h1>
  108. 108 
  109. 109 <p>Before accepting any data into your application, whether it be POST data from a form submission, COOKIE data, URI data,
  110. 110 XML-RPC data, or even data from the SERVER array, you are encouraged to practice this three step approach:</p>
  111. 111 
  112. 112 <ol>
  113. 113 <li>Filter the data as if it were tainted.</li>
  114. 114 <li>Validate the data to ensure it conforms to the correct  type, length, size, etc. (sometimes this step can replace step one)</li>
  115. 115 <li>Escape the data before submitting it into your database.</li>
  116. 116 </ol>
  117. 117 
  118. 118 <p>CodeIgniter provides the following functions to assist in this process:</p>
  119. 119 
  120. 120 <ul>
  121. 121 
  122. 122 <li><h2>XSS Filtering</h2>
  123. 123 
  124. 124 <p>CodeIgniter comes with a Cross Site Scripting filter.  This filter looks for commonly
  125. 125 used techniques to embed malicious Javascript into your data, or other types of code that attempt to hijack cookies
  126. 126 or do other malicious things. The XSS Filter is described <a href="../libraries/security.html">here</a>.
  127. 127 </p>
  128. 128 </li>
  129. 129 
  130. 130 <li><h2>Validate the data</h2>
  131. 131 
  132. 132 <p>CodeIgniter has a <a href="../libraries/form_validation.html">Form Validation Class</a> that assists you in validating, filtering, and prepping
  133. 133 your data.</p>
  134. 134 </li>
  135. 135 
  136. 136 <li><h2>Escape all data before database insertion</h2>
  137. 137 
  138. 138 <p>Never insert information into your database without escaping it. Please see the section that discusses
  139. 139 <a href="../database/queries.html">queries</a> for more information.</p>
  140. 140 
  141. 141 </li>
  142. 142 
  143. 143 </ul>
