  103. 103  <h2><a name="file_format"></a>File Format</h2>
  104. 104  <div class="guidelineDetails">
  105. 105  <p>Files should be saved with Unicode (UTF-8) encoding.  The <abbr title="Byte Order Mark">BOM</abbr>
  106. 106  should <em>not</em> be used.  Unlike UTF-16 and UTF-32, there's no byte order to indicate in
  107. 107  a UTF-8 encoded file, and the <abbr title="Byte Order Mark">BOM</abbr> can have a negative side effect in PHP of sending output,
  108. 108  preventing the application from being able to set its own headers.  Unix line endings should
  109. 109  be used (LF).</p>
  110. 110 
  111. 111  <p>Here is how to apply these settings in some of the more common text editors.  Instructions for your
  112. 112  text editor may vary; check your text editor's documentation.</p>
  113. 113 
  114. 114  <h5>TextMate</h5>
  115. 115 
  116. 116  <ol>
  117. 117  <li>Open the Application Preferences</li>
  118. 118  <li>Click Advanced, and then the "Saving" tab</li>
  119. 119  <li>In "File Encoding", select "UTF-8 (recommended)"</li>
  120. 120  <li>In "Line Endings", select "LF (recommended)"</li>
  121. 121  <li><em>Optional:</em> Check "Use for existing files as well" if you wish to modify the line
  122. 122  endings of files you open to your new preference.</li>
  123. 123  </ol>
  124. 124 
  125. 125  <h5>BBEdit</h5>
  126. 126 
  127. 127  <ol>
  128. 128  <li>Open the Application Preferences</li>
  129. 129  <li>Select "Text Encodings" on the left.</li>
  130. 130  <li>In "Default text encoding for new documents", select "Unicode (UTF-8, no BOM)"</li>
  131. 131  <li><em>Optional:</em> In "If file's encoding can't be guessed, use", select
  132. 132  "Unicode (UTF-8, no BOM)"</li>
  133. 133  <li>Select "Text Files" on the left.</li>
  134. 134  <li>In "Default line breaks", select "Mac OS X and Unix (LF)"</li>
  135. 135  </ol>
  136. 136  </div>
  137. 137 
  138. 138  <h2><a name="php_closing_tag"></a>PHP Closing Tag</h2>
  139. 139  <div class="guidelineDetails">
  140. 140  <p>The PHP closing tag on a PHP document <strong>?&gt;</strong> is optional to the PHP parser.  However, if used, any whitespace following the closing tag, whether introduced
  141. 141  by the developer, user, or an FTP application, can cause unwanted output, PHP errors, or if the latter are suppressed, blank pages.  For this reason, all PHP files should
  142. 142  <strong>OMIT</strong> the closing PHP tag, and instead use a comment block to mark the end of file and it's location relative to the application root.
  143. 143  This allows you to still identify a file as being complete and not truncated.</p>
  144. 144 <code><strong>INCORRECT</strong>:
  145. 145 &lt;?php
  146. 146 
  147. 147 echo "Here's my code!";
  148. 148 
  149. 149 ?&gt;
  150. 150 
  151. 151 <strong>CORRECT</strong>:
  152. 152 &lt;?php
  153. 153 
  154. 154 echo "Here's my code!";
  155. 155 
  156. 156 /* End of file myfile.php */
  157. 157 /* Location: ./system/modules/mymodule/myfile.php */
  158. 158 </code>
  159. 159  </div>

  162. 162  <h2><a name="class_and_method_naming"></a>Class and Method Naming</h2>
  163. 163  <div class="guidelineDetails">
  164. 164  <p>Class names should always start with an uppercase letter.  Multiple words should be separated with an underscore, and not CamelCased.  All other class methods should be entirely lowercased and named to clearly indicate their function, preferably including a verb.  Try to avoid overly long and verbose names.</p>
  165. 165 
  166. 166  <code><strong>INCORRECT</strong>:
  167. 167 class superclass
  168. 168 class SuperClass
  169. 169 
  170. 170 <strong>CORRECT</strong>:
  171. 171 class Super_class</code>
  172. 172 
  173. 173 
  174. 174  <code>class Super_class {
  175. 175 
  176. 176  function __construct()
  177. 177  {
  178. 178 
  179. 179  }
  180. 180 }</code>
  181. 181 
  182. 182  <p>Examples of improper and proper method naming:</p>
  183. 183 
  184. 184  <code><strong>INCORRECT</strong>:
  185. 185 function fileproperties() // not descriptive and needs underscore separator
  186. 186 function fileProperties() // not descriptive and uses CamelCase
  187. 187 function getfileproperties() // Better!  But still missing underscore separator
  188. 188 function getFileProperties() // uses CamelCase
  189. 189 function get_the_file_properties_from_the_file() // wordy
  190. 190 
  191. 191 <strong>CORRECT</strong>:
  192. 192 function get_file_properties() // descriptive, underscore separator, and all lowercase letters</code>
  193. 193 
  194. 194  </div>
  195. 195 
  196. 196 
  197. 197  <h2><a name="variable_names"></a>Variable Names</h2>
  198. 198  <div class="guidelineDetails">
  199. 199  <p>The guidelines for variable naming is very similar to that used for class methods.  Namely, variables should contain only lowercase letters, use underscore separators, and be reasonably named to indicate their purpose and contents. Very short, non-word variables should only be used as iterators in for() loops.</p>
  200. 200 <code><strong>INCORRECT</strong>:
  201. 201 $j = &apos;foo&apos;; // single letter variables should only be used in for() loops
  202. 202 $Str // contains uppercase letters
  203. 203 $bufferedText // uses CamelCasing, and could be shortened without losing semantic meaning
  204. 204 $groupid // multiple words, needs underscore separator
  205. 205 $name_of_last_city_used // too long
  206. 206 
  207. 207 <strong>CORRECT</strong>:
  208. 208 for ($j = 0; $j &lt; 10; $j++)
  209. 209 $str
  210. 210 $buffer
  211. 211 $group_id
  212. 212 $last_city
  213. 213 </code>
  214. 214  </div>
  215. 215 
  216. 216 
  217. 217  <h2><a name="commenting"></a>Commenting</h2>
  218. 218  <div class="guidelineDetails">
  219. 219  <p>In general, code should be commented prolifically.  It not only helps describe the flow and intent of the code for less experienced programmers, but can prove invaluable when returning to your own code months down the line.  There is not a required format for comments, but the following are recommended.</p>
  220. 220 
  221. 221  <p><a href="http://manual.phpdoc.org/HTMLSmartyConverter/HandS/phpDocumentor/tutorial_phpDocumentor.howto.pkg.html#basics.docblock">DocBlock</a> style comments preceding class and method declarations so they can be picked up by IDEs:</p>
  222. 222 
  223. 223 <code>/**
  224. 224  * Super Class
  225. 225  *
  226. 226  * @package Package Name
  227. 227  * @subpackage Subpackage
  228. 228  * @category Category
  229. 229  * @author Author Name
  230. 230  * @link http://example.com
  231. 231  */
  232. 232 class Super_class {</code>
  233. 233 
  234. 234 <code>/**
  235. 235  * Encodes string for use in XML
  236. 236  *
  237. 237  * @access public
  238. 238  * @param string
  239. 239  * @return string
  240. 240  */
  241. 241 function xml_encode($str)</code>
  242. 242 
  243. 243  <p>Use single line comments within code, leaving a blank line between large comment blocks and code.</p>
  244. 244 
  245. 245 <code>// break up the string by newlines
  246. 246 $parts = explode("\n", $str);
  247. 247 
  248. 248 // A longer comment that needs to give greater detail on what is
  249. 249 // occurring and why can use multiple single-line comments.  Try to
  250. 250 // keep the width reasonable, around 70 characters is the easiest to
  251. 251 // read.  Don't hesitate to link to permanent external resources
  252. 252 // that may provide greater detail:
  253. 253 //
  254. 254 // http://example.com/information_about_something/in_particular/
  255. 255 
  256. 256 $parts = $this->foo($parts);
  257. 257 </code>
  258. 258  </div>
  259. 259

  261. 261  <h2><a name="constants"></a>Constants</h2>
  262. 262  <div class="guidelineDetails">
  263. 263  <p>Constants follow the same guidelines as do variables, except constants should always be fully uppercase.  <em>Always use CodeIgniter constants when appropriate, i.e. SLASH, LD, RD, PATH_CACHE, etc.</em></p>
  264. 264 <code><strong>INCORRECT</strong>:
  265. 265 myConstant // missing underscore separator and not fully uppercase
  266. 266 N // no single-letter constants
  267. 267 S_C_VER // not descriptive
  268. 268 $str = str_replace('{foo}', 'bar', $str); // should use LD and RD constants
  269. 269 
  270. 270 <strong>CORRECT</strong>:
  271. 271 MY_CONSTANT
  272. 272 NEWLINE
  273. 273 SUPER_CLASS_VERSION
  274. 274 $str = str_replace(LD.'foo'.RD, 'bar', $str);
  275. 275 </code>
  276. 276  </div>

  279. 279  <h2><a name="true_false_and_null"></a>TRUE, FALSE, and NULL</h2>
  280. 280  <div class="guidelineDetails">
  281. 281  <p><strong>TRUE</strong>, <strong>FALSE</strong>, and <strong>NULL</strong> keywords should always be fully uppercase.</p>
  282. 282 <code><strong>INCORRECT</strong>:
  283. 283 if ($foo == true)
  284. 284 $bar = false;
  285. 285 function foo($bar = null)
  286. 286 
  287. 287 <strong>CORRECT</strong>:
  288. 288 if ($foo == TRUE)
  289. 289 $bar = FALSE;
  290. 290 function foo($bar = NULL)</code>
  291. 291  </div>
  292. 292

  294. 294 
  295. 295  <h2><a name="logical_operators"></a>Logical Operators</h2>
  296. 296  <div class="guidelineDetails">
  297. 297  <p>Use of <strong>||</strong> is discouraged as its clarity on some output devices is low (looking like the number 11 for instance).
  298. 298  <strong>&amp;&amp;</strong> is preferred over <strong>AND</strong> but either are acceptable, and a space should always precede and follow <strong>!</strong>.</p>
  299. 299 <code><strong>INCORRECT</strong>:
  300. 300 if ($foo || $bar)
  301. 301 if ($foo AND $bar)  // okay but not recommended for common syntax highlighting applications
  302. 302 if (!$foo)
  303. 303 if (! is_array($foo))
  304. 304 
  305. 305 <strong>CORRECT</strong>:
  306. 306 if ($foo OR $bar)
  307. 307 if ($foo && $bar) // recommended
  308. 308 if ( ! $foo)
  309. 309 if ( ! is_array($foo))
  310. 310 </code>
  311. 311  </div>
  312. 312

  315. 315  <h2><a name="comparing_return_values_and_typecasting"></a>Comparing Return Values and Typecasting</h2>
  316. 316  <div class="guidelineDetails">
  317. 317  <p>Some PHP functions return FALSE on failure, but may also have a valid return value of "" or 0, which would evaluate to FALSE in loose comparisons.  Be explicit by comparing the variable type when using these return values in conditionals to ensure the return value is indeed what you expect, and not a value that has an equivalent loose-type evaluation.</p>
  318. 318  <p>Use the same stringency in returning and checking your own variables.  Use <strong>===</strong> and <strong>!==</strong> as necessary.
  319. 319 
  320. 320 <code><strong>INCORRECT</strong>:
  321. 321 // If 'foo' is at the beginning of the string, strpos will return a 0,
  322. 322 // resulting in this conditional evaluating as TRUE
  323. 323 if (strpos($str, 'foo') == FALSE)
  324. 324 
  325. 325 <strong>CORRECT</strong>:
  326. 326 if (strpos($str, 'foo') === FALSE)
  327. 327 </code>
  328. 328 
  329. 329 <code><strong>INCORRECT</strong>:
  330. 330 function build_string($str = "")
  331. 331 {
  332. 332  if ($str == "") // uh-oh!  What if FALSE or the integer 0 is passed as an argument?
  333. 333  {
  334. 334 
  335. 335  }
  336. 336 }
  337. 337 
  338. 338 <strong>CORRECT</strong>:
  339. 339 function build_string($str = "")
  340. 340 {
  341. 341  if ($str === "")
  342. 342  {
  343. 343 
  344. 344  }
  345. 345 }</code>
  346. 346 
  347. 347  <p>See also information regarding <a href="http://us3.php.net/manual/en/language.types.type-juggling.php#language.types.typecasting">typecasting</a>, which can be quite useful.  Typecasting has a slightly different effect which may be desirable.  When casting a variable as a string, for instance, NULL and boolean FALSE variables become empty strings, 0 (and other numbers) become strings of digits, and boolean TRUE becomes "1":</p>
  348. 348 
  349. 349 <code>$str = (string) $str; // cast $str as a string</code>
  350. 350 
  351. 351  </div>

  354. 354  <h2><a name="debugging_code"></a>Debugging Code</h2>
  355. 355  <div class="guidelineDetails">
  356. 356  <p>No debugging code can be left in place for submitted add-ons unless it is commented out, i.e. no var_dump(), print_r(), die(), and exit() calls that were used while creating the add-on, unless they are commented out.</p>
  357. 357 
  358. 358 <code>// print_r($foo);</code>
  359. 359  </div>

  363. 363  <h2><a name="whitespace_in_files"></a>Whitespace in Files</h2>
  364. 364  <div class="guidelineDetails">
  365. 365  <p>No whitespace can precede the opening PHP tag or follow the closing PHP tag.  Output is buffered, so whitespace in your files can cause output to begin before CodeIgniter outputs its content, leading to errors and an inability for CodeIgniter to send proper headers.  In the examples below, select the text with your mouse to reveal the incorrect whitespace.</p>
  366. 366 
  367. 367  <p><strong>INCORRECT</strong>:</p>
  368. 368 <code>
  369. 369 &lt;?php
  370. 370  // ...there is whitespace and a linebreak above the opening PHP tag
  371. 371  // as well as whitespace after the closing PHP tag
  372. 372 ?&gt;
  373. 373 </code>
  374. 374  <p><strong>CORRECT</strong>:</p>
  375. 375 <code>&lt;?php
  376. 376  // this sample has no whitespace before or after the opening and closing PHP tags
  377. 377 ?&gt;</code>
  378. 378 
  379. 379  </div>
  380. 380 
  381. 381 
  382. 382  <h2><a name="compatibility"></a>Compatibility</h2>
  383. 383  <div class="guidelineDetails">
  384. 384  <p>Unless specifically mentioned in your add-on's documentation, all code must be compatible with PHP version 5.1+.  Additionally, do not use PHP functions that require non-default libraries to be installed unless your code contains an alternative method when the function is not available, or you implicitly document that your add-on requires said PHP libraries.</p>
  385. 385  </div>

  387. 387 
  388. 388 
  389. 389  <h2><a name="class_and_file_names_using_common_words"></a>Class and File Names using Common Words</h2>
  390. 390  <div class="guidelineDetails">
  391. 391  <p>When your class or filename is a common word, or might quite likely be identically named in another PHP script, provide a unique prefix to help prevent collision.  Always realize that your end users may be running other add-ons or third party PHP scripts.  Choose a prefix that is unique to your identity as a developer or company.</p>
  392. 392 
  393. 393 <code><strong>INCORRECT</strong>:
  394. 394 class Email pi.email.php
  395. 395 class Xml ext.xml.php
  396. 396 class Import mod.import.php
  397. 397 
  398. 398 <strong>CORRECT</strong>:
  399. 399 class Pre_email pi.pre_email.php
  400. 400 class Pre_xml ext.pre_xml.php
  401. 401 class Pre_import mod.pre_import.php
  402. 402 </code>
  403. 403  </div>
  404. 404 
  405. 405 
  406. 406  <h2><a name="database_table_names"></a>Database Table Names</h2>
  407. 407  <div class="guidelineDetails">
  408. 408  <p>Any tables that your add-on might use must use the 'exp_' prefix, followed by a prefix uniquely identifying you as the developer or company, and then a short descriptive table name.  You do not need to be concerned about the database prefix being used on the user's installation, as CodeIgniter's database class will automatically convert 'exp_' to what is actually being used.</p>
  409. 409 
  410. 410 <code><strong>INCORRECT</strong>:
  411. 411 email_addresses // missing both prefixes
  412. 412 pre_email_addresses // missing exp_ prefix
  413. 413 exp_email_addresses // missing unique prefix
  414. 414 
  415. 415 <strong>CORRECT</strong>:
  416. 416 exp_pre_email_addresses
  417. 417 </code>
  418. 418 
  419. 419  <p class="important"><strong>NOTE:</strong> Be mindful that MySQL has a limit of 64 characters for table names.  This should not be an issue as table names that would exceed this would likely have unreasonable names.  For instance, the following table name exceeds this limitation by one character.  Silly, no? <strong>exp_pre_email_addresses_of_registered_users_in_seattle_washington</strong>
  420. 420  </div>

  424. 424  <h2><a name="one_file_per_class"></a>One File per Class</h2>
  425. 425  <div class="guidelineDetails">
  426. 426  <p>Use separate files for each class your add-on uses, unless the classes are <em>closely related</em>.  An example of CodeIgniter files that contains multiple classes is the Database class file, which contains both the DB class and the DB_Cache class, and the Magpie plugin, which contains both the Magpie and Snoopy classes.</p>
  427. 427  </div>

  430. 430 
  431. 431  <h2><a name="whitespace"></a>Whitespace</h2>
  432. 432  <div class="guidelineDetails">
  433. 433  <p>Use tabs for whitespace in your code, not spaces.  This may seem like a small thing, but using tabs instead of whitespace allows the developer looking at your code to have indentation at levels that they prefer and customize in whatever application they use.  And as a side benefit, it results in (slightly) more compact files, storing one tab character versus, say, four space characters.</p>
  434. 434  </div>
  435. 435 
  436. 436 
  437. 437 
  438. 438  <h2><a name="line_breaks"></a>Line Breaks</h2>
  439. 439  <div class="guidelineDetails">
  440. 440  <p>Files must be saved with Unix line breaks.  This is more of an issue for developers who work in Windows, but in any case ensure that your text editor is setup to save files with Unix line breaks.</p>
  441. 441  </div>

  445. 445  <h2><a name="code_indenting"></a>Code Indenting</h2>
  446. 446  <div class="guidelineDetails">
  447. 447  <p>Use Allman style indenting.  With the exception of Class declarations, braces are always placed on a line by themselves, and indented at the same level as the control statement that "owns" them.</p>
  448. 448 
  449. 449 <code><strong>INCORRECT</strong>:
  450. 450 function foo($bar) {
  451. 451  // ...
  452. 452 }
  453. 453 
  454. 454 foreach ($arr as $key => $val) {
  455. 455  // ...
  456. 456 }
  457. 457 
  458. 458 if ($foo == $bar) {
  459. 459  // ...
  460. 460 } else {
  461. 461  // ...
  462. 462 }
  463. 463 
  464. 464 for ($i = 0; $i &lt; 10; $i++)
  465. 465  {
  466. 466  for ($j = 0; $j &lt; 10; $j++)
  467. 467  {
  468. 468  // ...
  469. 469  }
  470. 470  }
  471. 471 
  472. 472 <strong>CORRECT</strong>:
  473. 473 function foo($bar)
  474. 474 {
  475. 475  // ...
  476. 476 }
  477. 477 
  478. 478 foreach ($arr as $key => $val)
  479. 479 {
  480. 480  // ...
  481. 481 }
  482. 482 
  483. 483 if ($foo == $bar)
  484. 484 {
  485. 485  // ...
  486. 486 }
  487. 487 else
  488. 488 {
  489. 489  // ...
  490. 490 }
  491. 491 
  492. 492 for ($i = 0; $i &lt; 10; $i++)
  493. 493 {
  494. 494  for ($j = 0; $j &lt; 10; $j++)
  495. 495  {
  496. 496  // ...
  497. 497  }
  498. 498 }</code>
  499. 499  </div>

  502. 502  <h2><a name="bracket_spacing"></a>Bracket and Parenthetic Spacing</h2>
  503. 503  <div class="guidelineDetails">
  504. 504  <p>In general, parenthesis and brackets should not use any additional spaces.  The exception is that a space should always follow PHP control structures that accept arguments with parenthesis (declare, do-while, elseif, for, foreach, if, switch, while), to help distinguish them from functions and increase readability.</p>
  505. 505 
  506. 506 <code>INCORRECT:
  507. 507 $arr[ $foo ] = 'foo';
  508. 508 
  509. 509 CORRECT:
  510. 510 $arr[$foo] = 'foo'; // no spaces around array keys
  511. 511 
  512. 512 
  513. 513 INCORRECT:
  514. 514 function foo ( $bar )
  515. 515 {
  516. 516 
  517. 517 }
  518. 518 
  519. 519 CORRECT:
  520. 520 function foo($bar) // no spaces around parenthesis in function declarations
  521. 521 {
  522. 522 
  523. 523 }
  524. 524 
  525. 525 
  526. 526 INCORRECT:
  527. 527 foreach( $query->result() as $row )
  528. 528 
  529. 529 CORRECT:
  530. 530 foreach ($query->result() as $row) // single space following PHP control structures, but not in interior parenthesis
  531. 531 </code>
  532. 532  </div>

  559. 559  <h2><a name="php_errors"></a>PHP Errors</h2>
  560. 560  <div class="guidelineDetails">
  561. 561  <p>Code must run error free and not rely on warnings and notices to be hidden to meet this requirement.  For instance, never access a variable that you did not set yourself (such as $_POST array keys) without first checking to see that it isset().</p>
  562. 562 
  563. 563  <p>Make sure that while developing your add-on, error reporting is enabled for ALL users, and that display_errors is enabled in the PHP environment.  You can check this setting with:</p>
  564. 564 
  565. 565 <code>if (ini_get('display_errors') == 1)
  566. 566 {
  567. 567  exit "Enabled";
  568. 568 }</code>
  569. 569 
  570. 570  <p>On some servers where display_errors is disabled, and you do not have the ability to change this in the php.ini, you can often enable it with:</p>
  571. 571 
  572. 572 <code>ini_set('display_errors', 1);</code>
  573. 573 
  574. 574  <p class="important"><strong>NOTE:</strong> Setting the <a href="http://us.php.net/manual/en/ref.errorfunc.php#ini.display-errors">display_errors</a> setting with ini_set() at runtime is not identical to having it enabled in the PHP environment.  Namely, it will not have any effect if the script has fatal errors</p>

  610. 610  <h2><a name="strings"></a>Strings</h2>
  611. 611  <div class="guidelineDetails">
  612. 612  <p>Always use single quoted strings unless you need variables parsed, and in cases where you do need variables parsed, use braces to prevent greedy token parsing.  You may also use double-quoted strings if the string contains single quotes, so you do not have to use escape characters.</p>
  613. 613 
  614. 614 <code><strong>INCORRECT</strong>:
  615. 615 "My String" // no variable parsing, so no use for double quotes
  616. 616 "My string $foo" // needs braces
  617. 617 'SELECT foo FROM bar WHERE baz = \'bag\'' // ugly
  618. 618 
  619. 619 <strong>CORRECT</strong>:
  620. 620 'My String'
  621. 621 "My string {$foo}"
  622. 622 "SELECT foo FROM bar WHERE baz = 'bag'"</code>
  623. 623  </div>

  627. 627  <h2><a name="sql_queries"></a>SQL Queries</h2>
  628. 628  <div class="guidelineDetails">
  629. 629  <p>MySQL keywords are always capitalized: SELECT, INSERT, UPDATE, WHERE, AS, JOIN, ON, IN, etc.</p>
  630. 630 
  631. 631  <p>Break up long queries into multiple lines for legibility, preferably breaking for each clause.</p>
  632. 632 
  633. 633 <code><strong>INCORRECT</strong>:
  634. 634 // keywords are lowercase and query is too long for
  635. 635 // a single line (... indicates continuation of line)
  636. 636 $query = $this->db->query("select foo, bar, baz, foofoo, foobar as raboof, foobaz from exp_pre_email_addresses
  637. 637 ...where foo != 'oof' and baz != 'zab' order by foobaz limit 5, 100");
  638. 638 
  639. 639 <strong>CORRECT</strong>:
  640. 640 $query = $this->db->query("SELECT foo, bar, baz, foofoo, foobar AS raboof, foobaz
  641. 641  FROM exp_pre_email_addresses
  642. 642  WHERE foo != 'oof'
  643. 643  AND baz != 'zab'
  644. 644  ORDER BY foobaz
  645. 645  LIMIT 5, 100");</code>
