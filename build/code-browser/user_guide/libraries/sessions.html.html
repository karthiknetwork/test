  58.  58 
  59.  59 <h1>Session Class</h1>
  60.  60 
  61.  61 <p>The Session class permits you maintain a user's "state" and track their activity while they browse your site.
  62.  62 The Session class stores session information for each user as serialized (and optionally encrypted) data in a cookie.
  63.  63 It can also store the session data in a database table for added security, as this permits the session ID in the
  64.  64 user's cookie to be matched against the stored session ID. By default only the cookie is saved.  If you choose to
  65.  65 use the database option you'll need to create the session table as indicated below.
  66.  66 </p>
  67.  67 
  68.  68 <p class="important"><strong>Note:</strong> The Session class does <strong>not</strong> utilize native PHP sessions. It
  69.  69 generates its own session data, offering more flexibility for developers.</p>
  70.  70 
  71.  71 <p class="important"><strong>Note:</strong> Even if you are not using encrypted sessions, you must set
  72.  72 an <a href="./encryption.html">encryption key</a> in your config file which is used to aid in preventing session data manipulation.</p>
  73.  73 
  74.  74 <h2>Initializing a Session</h2>
  75.  75 
  76.  76 <p>Sessions will typically run globally with each page load, so the session class must either be
  77.  77 <a href="../general/libraries.html">initialized</a> in your
  78.  78 <a href="../general/controllers.html">controller</a> constructors, or it can be
  79.  79 <a href="../general/autoloader.html">auto-loaded</a> by the system.
  80.  80 For the most part the session class will run unattended in the background, so simply initializing the class
  81.  81 will cause it to read, create, and update sessions.</p>
  82.  82 
  83.  83 
  84.  84 <p>To initialize the Session class manually in your controller constructor, use the <dfn>$this->load->library</dfn> function:</p>
  85.  85 
  86.  86 <code>$this->load->library('session');</code>
  87.  87 <p>Once loaded, the Sessions library object will be available using: <dfn>$this->session</dfn></p>
  88.  88 
  89.  89 
  90.  90 <h2>How do Sessions work?</h2>
  91.  91 
  92.  92 <p>When a page is loaded, the session class will check to see if valid session data exists in the user's session cookie.
  93.  93 If sessions data does <strong>not</strong> exist (or if it has expired) a new session will be created and saved in the cookie.
  94.  94 If a session does exist, its information will be updated and the cookie will be updated. With each update, the session_id will be regenerated.</p>
  95.  95 
  96.  96 <p>It's important for you to understand that once initialized, the Session class runs automatically.  There is nothing
  97.  97 you need to do to cause the above behavior to happen.  You can, as you'll see below, work with session data or
  98.  98 even add your own data to a user's session, but the process of reading, writing, and updating a session is automatic.</p>
  99.  99 
  100. 100 
  101. 101 <h2>What is Session Data?</h2>
  102. 102 
  103. 103 <p>A <em>session</em>, as far as CodeIgniter is concerned, is simply an array containing the following information:</p>
  104. 104 
  105. 105 <ul>
  106. 106 <li>The user's unique Session ID (this is a statistically random string with very strong entropy, hashed with MD5 for portability, and regenerated (by default) every five minutes)</li>
  107. 107 <li>The user's IP Address</li>
  108. 108 <li>The user's User Agent data (the first 120 characters of the browser data string)</li>
  109. 109 <li>The "last activity" time stamp.</li>
  110. 110 </ul>
  111. 111 
  112. 112 <p>The above data is stored in a cookie as a serialized array with this prototype:</p>
  113. 113 
  114. 114 <code>[array]<br />
  115. 115 (<br />
  116. 116 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'session_id'&nbsp;&nbsp;&nbsp;&nbsp;=> random hash,<br />
  117. 117 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'ip_address'&nbsp;&nbsp;&nbsp;&nbsp;=> 'string - user IP address',<br />
  118. 118 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'user_agent'&nbsp;&nbsp;&nbsp;&nbsp;=> 'string - user agent data',<br />
  119. 119 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'last_activity' => timestamp<br />
  120. 120 )</code>
  121. 121 
  122. 122 <p>If you have the encryption option enabled, the serialized array will be encrypted before being stored in the cookie,
  123. 123 making the data highly secure and impervious to being read or altered by someone. More info regarding encryption
  124. 124 can be <a href="encryption.html">found here</a>, although the Session class will take care of initializing
  125. 125 and encrypting the data automatically.</p>
  126. 126 
  127. 127 <p>Note: Session cookies are only updated every five minutes by default to reduce processor load.  If you repeatedly reload a page
  128. 128 you'll notice that the "last activity" time only updates if five minutes or more has passed since the last time
  129. 129 the cookie was written. This time is configurable by changing the $config['sess_time_to_update'] line in your system/config/config.php file.</p>
  130. 130 
  131. 131 <h2>Retrieving Session Data</h2>
  132. 132 
  133. 133 <p>Any piece of information from the session array is available using the following function:</p>
  134. 134 
  135. 135 <code>$this->session->userdata('<samp>item</samp>');</code>
  136. 136 
  137. 137 <p>Where <samp>item</samp> is the array index corresponding to the item you wish to fetch.  For example, to fetch the session ID you
  138. 138 will do this:</p>
  139. 139 
  140. 140 <code>$session_id = $this->session->userdata('<samp>session_id</samp>');</code>
  141. 141 
  142. 142 <p><strong>Note:</strong> The function returns FALSE (boolean) if the item you are trying to access does not exist.</p>
  143. 143 
  144. 144 
  145. 145 <h2>Adding Custom Session Data</h2>
  146. 146 
  147. 147 <p>A useful aspect of the session array is that you can add your own data to it and it will be stored in the user's cookie.
  148. 148 Why would you want to do this?  Here's one example:</p>
  149. 149 
  150. 150 <p>Let's say a particular user logs into your site. Once authenticated,
  151. 151 you could add their username and email address to the session cookie, making that data globally available to you without
  152. 152 having to run a database query when you need it.</p>
  153. 153 
  154. 154 <p>To add your data to the session array involves passing an array containing your new data to this function:</p>
  155. 155 
  156. 156 <code>$this->session->set_userdata(<samp>$array</samp>);</code>
  157. 157 
  158. 158 <p>Where <samp>$array</samp> is an associative array containing your new data.  Here's an example:</p>
  159. 159 
  160. 160 
  161. 161 <p><code>$newdata = array(<br />
  162. 162  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'username'&nbsp; => 'johndoe',<br />
  163. 163  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'email'&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;=> 'johndoe@some-site.com',<br />
  164. 164  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;'logged_in' => TRUE<br />
  165. 165  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;);<br />
  166. 166  <br />
  167. 167  $this->session->set_userdata(<samp>$newdata</samp>);</code></p>
  168. 168 <p>If you want to add userdata one value at a time, set_userdata() also supports this syntax. </p>
  169. 169 <p><code>$this-&gt;session-&gt;set_userdata('some_name', 'some_value');</code></p>
  170. 170 <p class="important"><strong>Note:</strong> Cookies can only hold 4KB of data, so be careful not to exceed the capacity.  The
  171. 171 encryption process in particular produces a longer data string than the original so keep careful track of how much data you are storing.</p>
  172. 172 
  173. 173 <h2>Retrieving All Session Data</h2>
  174. 174 <p>An array of all userdata can be retrieved as follows:</p>
  175. 175 <code>$this-&gt;session-&gt;all_userdata()</code>
  176. 176 
  177. 177 <p>And returns an associative array like the following:</p>
  178. 178 
  179. 179 <pre>
  180. 180 Array
  181. 181 (
  182. 182     [session_id] => 4a5a5dca22728fb0a84364eeb405b601
  183. 183     [ip_address] => 127.0.0.1
  184. 184     [user_agent] => Mozilla/5.0 (Macintosh; U; Intel Mac OS X 10_6_7;
  185. 185     [last_activity] => 1303142623
  186. 186 )
  187. 187 </pre>
  188. 188 
  189. 189 
  190. 190 <h2>Removing Session Data</h2>
  191. 191 <p>Just as set_userdata() can be used to add information into a session, unset_userdata() can be used to remove it, by passing the session key. For example, if you wanted to remove 'some_name' from your session information: </p>
  192. 192 <p><code>$this-&gt;session-&gt;unset_userdata('some_name');</code></p>
  193. 193 <p>This function can also be passed an associative array of items to unset.</p>
  194. 194 <p><code>$array_items = array('username' => '', 'email' => '');<br />
  195. 195 <br />
  196. 196 $this-&gt;session-&gt;unset_userdata(<samp>$array_items</samp>);</code></p>
  197. 197 <h2>Flashdata</h2>
  198. 198 <p>CodeIgniter supports &quot;flashdata&quot;, or session data that will only be available for the next server request, and are then automatically cleared. These can be very useful, and are typically used for informational or status messages (for example: &quot;record 2 deleted&quot;).</p>
  199. 199 <p>Note: Flash variables are prefaced with &quot;flash_&quot; so avoid this prefix in your own session names.</p>
  200. 200 <p>To add flashdata:</p>
  201. 201 <p><code>$this-&gt;session-&gt;set_flashdata('item', 'value');</code></p>
  202. 202 <p>You can also pass an array to set_flashdata(), in the same manner as set_userdata(). </p>
  203. 203 <p>To read a flashdata variable:</p>
  204. 204 <p><code>$this-&gt;session-&gt;flashdata('item');</code></p>
  205. 205 <p>If you find that you need to preserve a flashdata variable through an additional request, you can do so using the keep_flashdata() function.</p>
  206. 206 <p><code>$this-&gt;session-&gt;keep_flashdata('item');</code></p>
  207. 207 <h2>Saving Session Data to a Database</h2>
  208. 208 <p>While the session data array stored in the user's cookie contains a Session ID,
  209. 209 unless you store session data in a database there is no way to validate it.  For some applications that require little or no
  210. 210 security, session ID validation may not be needed, but if your application requires security, validation is mandatory.  Otherwise, an old session
  211. 211 could be restored by a user modifying their cookies.</p>
  212. 212 
  213. 213 <p>When session data is available in a database, every time a valid session is found in the user's cookie, a database
  214. 214 query is performed to match it.  If the session ID does not match, the session is destroyed.  Session IDs can never
  215. 215 be updated, they can only be generated when a new session is created.</p>
  216. 216 
  217. 217 
  218. 218 <p>In order to store sessions, you must first create a database table for this purpose.  Here is the basic
  219. 219 prototype (for MySQL) required by the session class:</p>
  220. 220 
  221. 221 <textarea class="textarea" style="width:100%" cols="50" rows="10">
  222. 222 CREATE TABLE IF NOT EXISTS  `ci_sessions` (
  223. 223  session_id varchar(40) DEFAULT '0' NOT NULL,
  224. 224  ip_address varchar(45) DEFAULT '0' NOT NULL,
  225. 225  user_agent varchar(120) NOT NULL,
  226. 226  last_activity int(10) unsigned DEFAULT 0 NOT NULL,
  227. 227  user_data text NOT NULL,
  228. 228  PRIMARY KEY (session_id),
  229. 229  KEY `last_activity_idx` (`last_activity`)
  230. 230 );
  231. 231 </textarea>
  232. 232 
  233. 233 <p><strong>Note:</strong> By default the table is called <dfn>ci_sessions</dfn>, but you can name it anything you want
  234. 234 as long as you update the <kbd>application/config/config.php</kbd> file so that it contains the name you have chosen.
  235. 235 Once you have created your database table you can enable the database option in your config.php file as follows:</p>
  236. 236 
  237. 237 <code>$config['sess_use_database'] = TRUE;</code>
  238. 238 
  239. 239 <p>Once enabled, the Session class will store session data in the DB.</p>
  240. 240 
  241. 241 <p>Make sure you've specified the table name in your config file as well:</p>
  242. 242 
  243. 243 <code>$config['sess_table_name'] = 'ci_sessions';</code>
  244. 244 
  245. 245 <p class="important"><strong>Note:</strong> The Session class has built-in garbage collection which clears out expired sessions so you
  246. 246 do not need to write your own routine to do it.</p>
  247. 247 
  248. 248 
  249. 249 <h2>Destroying a Session </h2>
  250. 250 <p>To clear the current session: </p>
  251. 251 <code>$this-&gt;session-&gt;sess_destroy();</code>
  252. 252 <p class="important"><strong>Note:</strong> This function should be the last one called, and even flash variables will no longer be available.  If you only want some items destroyed and not all, use <dfn>unset_userdata()</dfn>.</p>
  253. 253 
  254. 254 
  255. 255

  292. 292  <td class="td"><strong>sess_use_database</strong></td>
